# システム同定(プラント特性の測定)

VRChat の入力軸(視点・移動)の応答特性を実機で測定し、`plant.json`(プラントモデル)として保存する手順。ゲイン調整や制御方式の比較は、このモデルを組み込んだ模擬プラント上で実機なしに行える([ゲイン調整](gain-tuning.md))。

## 同定するもの

| 項目     | 内容                                                           | 使い道                                                              |
| -------- | -------------------------------------------------------------- | ------------------------------------------------------------------- |
| 静特性   | 指令値 [-1,1] → 定常速度(軸ごとの折れ線)                       | 不感帯(視点軸は ≈0.50 に折れ点)や量子化の把握、模擬プラントの速度場 |
| むだ時間 | OSC 送信 → ゲーム反映 → 描画 → キャプチャ → デコードの合計遅延 | 発振のしやすさを決める最も重要なパラメータ                          |
| dt 系列  | HUD の新フレーム間隔の実測値の列(ばらつき・ドロップ込み)       | 模擬プラントを実機と同じフレーム間隔で駆動する                      |

対象軸と OSC 入力の対応:

| 軸        | `/input/`        | 単位  | 備考                                                                        |
| --------- | ---------------- | ----- | --------------------------------------------------------------------------- |
| `yaw`     | `LookHorizontal` | deg/s | + で右旋回。不感帯の折れ点(≈0.50)は二分探索で自動検出                       |
| `pitch`   | `LookVertical`   | deg/s | ±80° クランプ回避のため角度ガード付き(水平から ±`--pitch-span`)             |
| `forward` | `Vertical`       | m/s   | 前後移動                                                                    |
| `strafe`  | `Horizontal`     | m/s   | 左右移動。align(横移動での最終照準)の精度は、この軸の低指令域の特性で決まる |

## 測定の流れ(src/vrc_autopilot/sysid/adaptive.py)

1. cmd=1.0 で最大速度を測る
2. 折れ点(不感帯オンセット)を二分探索で `--onset-tol`(既定 0.01)まで絞る
3. 折れ点より上を数レベル測って傾きを取る
4. 固定レベルの 0→v 遷移を `--burst`(既定 40)回連射し、むだ時間を
   分布(中央値・ばらつき・最悪値。meta の `deadtime_stats`)として取る

プチフリ(HUD が一瞬飛ぶ)をフレーム間隔から検知したレベルは捨てて自動で
測り直す。同定側でもフリーズを含むセグメント/遷移は除外される(`--from-log`
にも効く)。

## 実機での測定手順

```bash
probe-axes                       # 4軸すべて → logs/probe_<日時>/
probe-axes --axes yaw,strafe     # 軸を絞る
probe-axes --max-travel 0.4      # 狭い場所向けに移動範囲を狭める
probe-axes --axes pitch --out logs/probe_XXXX   # 1軸だけ測り直して既存に合流
```

前提: VRChat が起動済みで、HUD(`PixelHUDEnable`)が表示され、OSC 受信が有効になっていること。既定設定なら数分程度で終わる。Ctrl-C で中断しても完了済みの軸だけで同定される。plant.json は出力ディレクトリに生ログがある軸すべてから組む(だから `--out` に既存ディレクトリを渡すと 1 軸だけ差し替わる)。

### 必要なスペース

- `yaw` / `pitch` はその場で回転するだけなので、スペースは不要。pitch は ±80° クランプに張り付かないよう、水平から ±`--pitch-span`(既定 70°)の角度ガード内で振る。
- `forward` / `strafe` は開始位置から軸方向に ±`--max-travel`(既定 3m)の範囲内を往復する。
- 範囲を狭くすると高速側(0.8–1.0)の定常区間が短くなって精度は落ちるが、微調整に効くのは低速側なので、実用上の影響は小さい。

### 主なオプション

| フラグ         | 既定 | 意味                              |
| -------------- | ---- | --------------------------------- |
| `--burst`      | 40   | むだ時間バーストの遷移回数        |
| `--onset-tol`  | 0.01 | 折れ点の探索分解能(指令値)        |
| `--pitch-span` | 70   | pitch を水平から振る角度幅[°]     |
| `--max-travel` | 3    | 移動範囲の片側幅[m]               |

## 出力

`logs/probe_<日時>/` に以下が保存される:

- `probe_<軸>.csv` / `segments_<軸>.csv` — 生ログ(サンプルの時系列と指令の送信時刻)
- `plant.json` — 同定結果(軸ごとの静特性の折れ線とむだ時間、それに dt 系列)
- `<軸>.png` — 静特性のプロット(測定点と補間線)。**まずこれを目で確認する**:
  - yaw の 0.50 付近に折れ点が出ているか
  - strafe / forward は指令に比例しているか、それとも歩き/走りの 2 段階に量子化されているか
  - 点のばらつきが大きい軸は測り直す

生ログさえ残っていれば、同定だけを何度でもやり直せる(実機は不要):

```bash
probe-axes --from-log logs/probe_XXXX
```

## しくみ(src/vrc_autopilot/sysid/identify.py)

- **プローブ** `run_axis_probe` / `run_pitch_probe` / `run_move_probe`: 階段状の指令列を送りながらポーズの時系列を記録する。移動軸は開始位置からの変位(軸方向への射影)、pitch はセグメント開始からの振れ幅で指令を切り返す。ガード付きセグメント中に HUD が途絶したときは、ガードを見られないまま動き続けないよう指令を 0 に戻して打ち切る。`PoseSource` と送信コールバック、クロックの抽象にしか依存しないので、模擬プラントと模擬クロックを渡せばヘッドレスでも動く(`tests/test_sysid.py`)。
- **静特性** `identify_axis`: 各セグメントの後半から応答の傾きを最小二乗で求め、同一レベルの結果は平均する(先頭の `max(40%, 0.2s)` は過渡やむだ時間、折り返し直後の逆走を含むため捨てる)。pitch はクランプ(実測 ±80°)に張り付いたサンプルを除外する。移動軸は、セグメント開始時の体の向きを基準に変位を前方向/右方向へ射影する(仮に VRChat の +Horizontal が左向きだったとしても、符号も含めて正しく同定できる)。同定に失敗した軸はスキップする。
- **むだ時間**: 0→v の遷移で応答がしきい値(角度 0.1° / 位置 0.01m)を最初に超えたサンプル (t, r) から、定常速度で `t - r/steady` へ逆外挿して動き出し時刻を求め、送信時刻との差の中央値を採用する。分布(mean/std/p95/max)は `plant.json` の meta に残る。
- **プチフリ耐性**: フレーム間隔中央値の 2.5 倍を超えるギャップを HUD 途絶とみなし、定常窓に含むセグメントを静特性から、交差が跨いだ遷移をむだ時間から除外する。プローブ時点でも検知して測り直す。
- **モデル** `AxisModel.rate(cmd)` は測定点の線形補間(範囲外は端の値で頭打ち)。不感帯オンセットは `AxisModel.onset` が測定点から導出する(loop_analysis の実効ゲインもこれを使う)。`PlantModel` が各軸のモデルと dt 系列をまとめて JSON に保存する。

## 限界

- モデル化しているのは静特性とむだ時間、dt 分布だけで、速度ランプは表現できない。ただし実測にランプは無い(0→v 遷移で速度が1サンプルで定常値に飛ぶ)ので影響しない。
- 特性はワールド側の設定(移動速度など)に依存する。環境が変わったら測り直すこと。
