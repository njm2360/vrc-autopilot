# システム同定(プラント特性の測定)

VRChat の入力軸(視点・移動)の応答特性を実機で測定し、`plant.json`
(プラントモデル)として保存する手順。ゲイン調整や制御方式の比較は、この
モデルを組み込んだ模擬プラント上で実機なしに行える([ゲイン調整](gain-tuning.md))。

## 同定するもの

| 項目     | 内容                                                           | 使い道                                                              |
| -------- | -------------------------------------------------------------- | ------------------------------------------------------------------- |
| 静特性   | 指令値 [-1,1] → 定常速度(軸ごとの折れ線)                       | 不感帯(視点軸は ≈0.55 に折れ点)や量子化の把握、模擬プラントの速度場 |
| むだ時間 | OSC 送信 → ゲーム反映 → 描画 → キャプチャ → デコードの合計遅延 | 発振のしやすさを決める最も重要なパラメータ                          |
| dt 系列  | HUD の新フレーム間隔の実測値の列(ばらつき・ドロップ込み)       | 模擬プラントを実機と同じフレーム間隔で駆動する                      |

対象軸と OSC 入力の対応:

| 軸        | `/input/`        | 単位  | 備考                                                                       |
| --------- | ---------------- | ----- | -------------------------------------------------------------------------- |
| `yaw`     | `LookHorizontal` | deg/s | + で右旋回。不感帯の折れ点付近(0.45–0.7)を細かく測る                       |
| `pitch`   | `LookVertical`   | deg/s | ±90° で頭打ちになるため保持時間は短め(`--pitch-hold`)                      |
| `forward` | `Vertical`       | m/s   | 前後移動                                                                   |
| `strafe`  | `Horizontal`     | m/s   | 左右移動。ストレイフ微調整がうまくいくかは、この軸の低指令域の特性で決まる |

## 実機での測定手順

```
probe-axes                       # 4軸すべて → logs/probe_<日時>/
probe-axes --axes yaw,strafe     # 軸を絞る
probe-axes --max-travel 0.4      # 狭い場所向けに移動範囲を狭める
```

前提: VRChat が起動済みで、HUD(`HUD_Enable`)が表示され、OSC 受信が有効に
なっていること。既定設定なら数分程度で終わる。

**必要なスペース**

- `yaw` / `pitch` はその場で回転するだけなので、スペースは不要。
- `forward` / `strafe` は開始位置から軸方向に ±`--max-travel`(既定 3m)の範囲内を往復する。
- 範囲を狭くすると高速側(0.8–1.0)の定常区間が短くなって精度は落ちるが、
  微調整に効くのは低速側なので、実用上の影響は小さい。

**主なオプション**

| フラグ                    | 既定               | 意味                                                                   |
| ------------------------- | ------------------ | ---------------------------------------------------------------------- |
| `--levels`                | 0.01〜1.0 の1%刻み | 視点軸に与える指令レベル                                               |
| `--move-levels`           | 0.01〜1.0 の1%刻み | 移動軸に与える指令レベル                                               |
| `--hold` / `--pitch-hold` | 1.0 / 0.5          | 視点軸で各レベルを保持する時間[s]                                      |
| `--move-hold`             | 1.2                | 移動軸の片道あたりの上限時間[s] (位置ガードが先に働けば途中で打ち切る) |
| `--max-travel`            | 3                  | 移動範囲の片側幅[m]                                                    |
| `--passes`                | 1                  | 移動軸で 1 レベルあたり往復する回数                                    |
| `--settle`                | 0.6                | レベル間で指令 0 を保持する時間[s] (むだ時間は 0→v の遷移で測る)       |

## 出力

`logs/probe_<日時>/` に以下が保存される:

- `probe_<軸>.csv` / `segments_<軸>.csv` — 生ログ(サンプルの時系列と指令の送信時刻)
- `plant.json` — 同定結果(軸ごとの静特性の折れ線・むだ時間、dt 系列)
- `<軸>.png` — 静特性のプロット(測定点と補間線)。**まずこれを目で確認する**:
  - yaw の 0.55 付近に折れ点が出ているか
  - strafe / forward は指令に比例しているか、それとも歩き/走りの 2 段階に量子化されているか
  - 点のばらつきが大きい軸は、保持時間を延ばして測り直す

生ログさえ残っていれば、同定だけを何度でもやり直せる(実機は不要):

```
probe-axes --from-log logs/probe_XXXX
```

## しくみ(pose_hud/sysid.py)

- **プローブ** `run_axis_probe` / `run_move_probe`: 階段状の指令列を送りながら
  ポーズの時系列を記録する。移動軸版は開始位置からの変位(軸方向への射影)を
  監視し、範囲の端で折り返す。`PoseSource`・送信コールバック・クロックの
  抽象にしか依存しないので、模擬プラントと模擬クロックを渡せばヘッドレスでも
  動く(`tests/test_sysid.py` では、特性が既知の合成プラントをプローブ → 同定
  して、元の特性と一致することを確認している)。
- **静特性** `identify_axis`: 各セグメントの後半から応答の傾きを最小二乗で求め、
  同一レベルの結果は平均する(先頭の `max(40%, 0.2s)` は過渡・むだ時間・
  折り返し直後の逆走を含むため捨てる)。移動軸は、セグメント開始時の体の向きを
  基準に変位を前方向/右方向へ射影する(仮に VRChat の +Horizontal が左向き
  だったとしても、符号も含めて正しく同定できる)。
- **むだ時間**: 0→v の遷移で応答がしきい値(角度 0.1° / 位置 0.01m)を超えた
  時刻から、しきい値に達するまでの所要時間(しきい値/速度)を引いて逆算し、
  全遷移の中央値を採用する。
- **モデル** `AxisModel.rate(cmd)` は測定点の線形補間(範囲外は端の値で頭打ち)。
  `PlantModel` が各軸のモデルと dt 系列をまとめて JSON に保存する。

## 限界

- モデル化しているのは**静特性・むだ時間・dt 分布**だけ。視点軸に時間方向の
  平滑化(指令が一定でも角速度が立ち上がるまでランプする挙動)があっても、
  モデルには反映されない。疑わしい場合は生 CSV でステップ応答を確認し、顕著で
  あれば一次遅れの追加が必要になる。
- 特性はワールド側の設定(移動速度など)に依存する。環境が変わったら測り直すこと。
