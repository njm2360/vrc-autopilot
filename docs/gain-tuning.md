# ゲイン調整メモ

巡回制御(`PatrolGains`)のチューニング手順。**同定済みプラント(実機なし)でも
実機でも同じゲインフラグ・同じ指標で回せる**ようにしてある。基本の流れは:

```
1. probe-axes で plant.json を作る(実機で一度だけ。手順は system-identification.md)
2. sim-face でゲインを探索する(実機なし、何度でも)
3. patrol-buttons で実機に持ち込み、ログの指標で答え合わせする
```

シミュレータと実機で結果が食い違ったら、ゲインではなくモデルを疑う
(静特性プロットとむだ時間を見直す)。

## ゲインの意味と効き方

ゲインの定義は `app/control/controller.py` の `PatrolGains` に集約されている。
CLI フラグは `sim-face` / `patrol-buttons` で共通(既定値も同じ)。

### 正対(face)yaw — 振動が出やすい軸

| フラグ            | 既定  | 効き方                                                                               |
| ----------------- | ----- | ------------------------------------------------------------------------------------ |
| `--turn-kp`       | 0.08  | 比例項。大きいほど速く向くが行き過ぎやすい                                           |
| `--turn-ki`       | 0.01  | 積分項。定常偏差を消す。溜まりすぎは `--turn-ilim`(0.5)で制限                        |
| `--turn-kd`       | 0.006 | ブレーキ。むだ時間が大きいと効きが遅れて逆効果になることも                           |
| `--turn-deadzone` | 0.55  | **不感帯補償**。非ゼロ指令をこの値まで底上げして、視点軸の反応しない範囲を飛び越える |
| `--face-tol`      | 1.0   | 収束とみなす角度[deg]。この範囲内は指令0                                             |
| `--settle`        | 3     | 収束判定に必要な連続フレーム数                                                       |

**既知の問題**: 不感帯補償により最小指令が ±0.55 に飛ぶため、tol 境界付近では
最小の動きでも tol を跨いでしまい、反対向きに戻してはまた跨ぐ、という
リミットサイクル(振動)になりやすい。むだ時間が長いほど悪化する。合成プラントでの再現例: 既定ゲインで
osc 18〜19 回・オーバーシュート ≈7°・未収束。対策の候補:

- `--face-tol` を広げる(必要な精度との兼ね合い)
- `--turn-deadzone` を実測の折れ点ぎりぎりまで下げる(plant.json の yaw カーブで
  「速度が立ち上がる直前」の指令値を読む)
- kp/ki を下げて、指令が補償で 0.55 に底上げされる頻度を減らす
- (対応済み)look 軸ではなく体の横移動で残りの yaw 誤差を吸収する(align フェーズ)

### 正対(face)pitch

`--pitch-kp/-ki/-kd/-ilim/-deadzone`。既定は補償なし(0)。上下がなかなか
合わないときだけ `--pitch-deadzone 0.5` 程度から試す。

### 移動追従(nav)

| フラグ                  | 既定              | 効き方                                                                 |
| ----------------------- | ----------------- | ---------------------------------------------------------------------- |
| `--nav-turn-kp/-ki/-kd` | 0.035/0.015/0.004 | 進行方向 yaw。不感帯補償はなし(小さい指令が遊びとして働き、暴れを防ぐ) |
| `--speed`               | 0.7               | 巡航前進の上限                                                         |
| `--fwd-kp/-kd`          | 1.5/0.1           | 最終ウェイポイントの減速(誤差=距離[m])                                 |
| `--arrive`              | 0.35              | 到達半径[m]                                                            |

## 応答指標(AxisMetrics)の読み方

`sim-face` の表も実機ログも同じ計算で出している(`app/control/telemetry.py`)。

| 指標           | 意味                                    | 見方                                         |
| -------------- | --------------------------------------- | -------------------------------------------- |
| `osc`          | 誤差の符号反転回数                      | **振動の主指標**。2〜3 以下が目安、2桁は発振 |
| `overshoot`    | 目標を行き過ぎた量(反対側のピーク)      | tol より大きいと行き過ぎて往復している       |
| `settle_time`  | tol 内を維持し始めた時刻                | None は未収束                                |
| `iae` / `itae` | 誤差の積分(itae は収束が遅いほど大きい) | ゲイン候補同士の総合比較に                   |
| `effort`       | 指令量の積分                            | 同成績なら小さいほうが滑らか                 |

## A. 同定プラントで回す(実機なし)

```
sim-face --model logs/probe_XXXX/plant.json
sim-face --model ... --turn-kp 0.05 --turn-deadzone 0.50 --yaw-err 30,10,5,2
sim-face --model ... --pitch-err 5            # pitch も同時制御して検証
sim-face --model ... --log logs/simface.csv   # フレーム記録も残す
```

- 本番と同一の正対ループ(`turn_to` → `_face_loop`)を実時間で回す。模擬プラント
  (`app/sysid/simplant.py`)は同定した静特性・むだ時間・dt 系列に従って状態を
  積分する。
- `--yaw-err` の小さい値(2〜5°)が、最後のボタン合わせに相当する。**まず現行ゲインで
  振動が再現されることを確認**してから(モデルの妥当性確認)、ゲインを振る。
- 探索のコツ: 1変数ずつ。deadzone → kp → ki → kd の順が事故が少ない。
  候補が絞れたら `--yaw-err` を増やして大誤差からの引き込みも確認する。
- スクリプトから回したい場合は `SimulatedVRChat(plant).start_realtime()` を
  `turn_to` / `aim_at` / `follow_path` にそのまま渡せる(必要な 3 つの
  インターフェースをすべて実装している)。`tests/test_sysid.py` に例がある。

## B. 実機で回す

```
patrol-buttons --map <map.npz> --target x,y,z,face_yaw --turn-kp 0.05 --turn-deadzone 0.50
```

- ゲインフラグは sim-face と同名。`logs/patrol_<日時>.csv` にフレーム記録が残る。
- 実行後、正対フェーズ(`phase=face`)の `yaw_err` / `turn` 列を見る。振動して
  いれば `yaw_err` が tol 前後で符号を往復し、`turn` が ±0.55 の間を行き来して
  いるはず。`turn_p/turn_i/turn_d` の内訳でどの項が暴れているかも分かる。
- `aim` の戻り値(`AimResult.yaw/pitch`)にも同じ AxisMetrics が入るので、
  シミュレータの表とそのまま突き合わせられる。
- 視点をマウス入力(`--look mouse`)にする場合は不感帯が無いので
  `--turn-deadzone 0` にすること(ゲインは OSC とは別物。要再調整)。

## 既定値の由来

- `turn_deadzone: 0.55` — 視点軸は指令 0.55 以下ではほとんど反応せず、超えると
  急に速くなる、という実測に基づく。正確な折れ点は plant.json を測り直して
  更新すること。
- nav 側に補償が無いのは、連続追従では小さい指令の不感帯が自然な遊びとして
  働くため(補償を入れると暴れる)。
